# Generated Data Contract

`site/data` is the production API for the static frontend. The Go builder writes these files locally, and GitHub Pages serves the committed bytes without a server.

## Versioning

Current contract versions:

- `dataContractVersion`: `1`
- `schemaVersion`: `1`

`catalogue.json` and `build_manifest.json` include these fields directly. Array-shaped JSON files keep their top-level array shape for frontend compatibility; their schema version is recorded in `build_manifest.json.generatedFiles`.

Non-breaking changes may add optional fields or new generated files. A breaking change is any removal, rename, type change, required-field change, or top-level shape change for an existing file. To make one intentionally, update the Go constants, exporter tests/goldens, this document, smoke checks, frontend loading code, and the readiness checklist in the same change.

## Manifest And Checksums

`site/data/build_manifest.json` is the authoritative metadata file. Important fields:

- `dataContractVersion`, `schemaVersion`: global data contract identifiers.
- `builtAt`: builder timestamp in RFC3339.
- `sourceMode`, `trading212Environment`, `trading212BaseUrl`, `trading212FetchAt`: source provenance.
- Count fields such as `instrumentCount`, `securityCount`, `companyCount`, `listingCount`, `themeCount`, `exposureCount`, `relationshipCount`, and review/failure counts.
- `rawSnapshots`, `trading212HttpDiagnostics`, `trading212RateLimits`: replay and live fetch diagnostics when available.
- `dataFreshness`: coarse age of the source snapshot.
- `generatedFiles`: ordered metadata for every generated `site/data` file.

Each `generatedFiles` row has:

- `path`: canonical static path, such as `site/data/catalogue.json`.
- `format`: `json` or `csv`.
- `schemaVersion`: file schema version.
- `sha256`: SHA-256 checksum.
- `bytes`: generated byte length.
- `checksumMode`: optional convention for files whose checksum is not the literal file bytes.

For all files except `build_manifest.json`, `sha256` is the checksum of the exact generated file bytes. For `build_manifest.json`, `sha256` is computed from a stable manifest projection with `generatedFiles` removed, while `bytes` is the final generated file length. Its `checksumMode` is `projection_excludes_generatedFiles`. This avoids a self-referential checksum loop while still making manifest content changes visible.

`catalogue.json.manifest` intentionally omits `generatedFiles`; use standalone `build_manifest.json` for checksum verification.

## Frontend Files

These files are intended for frontend loading:

- `catalogue.json`: complete bundle used by `site/assets/app.js`.
- `search_index.json`: compact searchable documents.
- `companies.json`, `sectors.json`, `industries.json`, `themes.json`, `supply_chains.json`: standalone slices for future frontend code.
- `securities.json`, `listings.json`, `relationships.json`: standalone normalized identity and manual relationship slices.
- `build_manifest.json`: metadata for review/export tooling.

These files are primarily for human review and spreadsheet workflows:

- `tickers.csv`
- `securities.csv`
- `listings.csv`
- `unclassified.csv`
- `identity_issues.csv`
- `enrichment_failures.csv`

## JSON Files

### catalogue.json

Top-level object. Stable keys include `dataContractVersion`, `schemaVersion`, `generatedAt`, `manifest`, `tickers`, `securities`, `listings`, `companies`, `sectors`, `industries`, `themes`, `supplyChains`, `exposures`, `relationships`, `notes`, `unclassified`, and optional `identityIssues`.

The frontend depends on the existing object shape and direct arrays such as `catalogue.tickers`, `catalogue.themes`, and `catalogue.supplyChains`. Do not wrap this file in a `data` envelope without a compatible frontend migration.

### build_manifest.json

Top-level object using the manifest fields described above. This is the only file whose `generatedFiles` entry uses `checksumMode: "projection_excludes_generatedFiles"`.

### companies.json

Top-level array of company objects. Important fields include `id`, `name`, `primaryTicker`, `sector`, `industry`, `country`, `yahooSymbol`, `marketCap`, `securityIds`, `listingIds`, `tickerIds`, `themeIds`, `layerIds`, `relatedTickers`, `sources`, `identityConfidence`, `identityReasons`, `lastReviewed`, and `lastRefreshed`.

### securities.json

Top-level array matching `catalogue.securities`. Important fields include `id`, `isin`, `name`, `type`, `instrumentCategory`, `structureFlags`, `companyId`, `listingIds`, `tickerIds`, `currencySet`, `identityConfidence`, and `identityReasons`.

### listings.json

Top-level array matching `catalogue.listings`. Important fields include `id`, `ticker`, `securityId`, `companyId`, `exchangeCode`, `exchangeName`, and `currencyCode`.

### sectors.json

Top-level array of group counts with `id`, `name`, `count`, and optional `tickers`.

### industries.json

Top-level array of group counts with `id`, `name`, `count`, and optional `tickers`.

### themes.json

Top-level array of manual themes with `id`, `name`, optional `description`, and optional `color`.

### supply_chains.json

Top-level array of supply-chain definitions with `themeId`, `name`, optional `description`, and `layers`. Each layer has `id`, `name`, optional `description`, and `order`.

### search_index.json

Top-level array of search documents with `id`, `type`, `title`, optional `subtitle`, `text`, and optional `tickers`.

### relationships.json

Top-level array matching `catalogue.relationships`. Rows are sorted deterministically. Fields:

- `relationshipType`: one of `peer`, `substitute`, `upstream_supplier`, `downstream_customer`, or `related_play`.
- `sourceTicker`, `sourceIsin`, `sourceCompanyId`: exactly one source identifier.
- `targetTicker`, `targetIsin`, `targetCompanyId`: exactly one target identifier.
- `themeId`, `layerId`: optional theme/layer context; `layerId` requires `themeId`.
- `confidence`: manual or rule confidence value.
- `sourceUrl`: reviewed source URL.
- `rationale`: optional review note.
- `lastReviewed`: `YYYY-MM-DD`.

### sources.json

No standalone `sources.json` is generated in contract V1. Entity-local `sources` arrays on tickers and companies are the stable contract. Global source reuse is deferred until the project has stable source IDs instead of only URL/kind/label tuples.

## CSV Files

CSV headers are stable contract fields. Add new columns at the end when possible. If a header is renamed, removed, or reordered, treat it as a breaking schema change.

### tickers.csv

Header: `ticker,name,isin,company_id,security_id,type,instrument_category,structure_flags,currency,exchange,yahoo_symbol,sector,industry,country,market_cap,directionality,identity_confidence,identity_reasons,themes,layers,unclassified`

Broker-level investable row. Semicolon-delimited list fields are `structure_flags`, `identity_reasons`, `themes`, and `layers`.

### securities.csv

Header: `security_id,isin,name,type,instrument_category,structure_flags,company_id,listing_ids,ticker_ids,currency_set,identity_confidence,identity_reasons`

Normalized security identity row. Semicolon-delimited list fields are `structure_flags`, `listing_ids`, `ticker_ids`, `currency_set`, and `identity_reasons`.

### listings.csv

Header: `listing_id,ticker,security_id,company_id,exchange_code,exchange_name,currency_code`

Broker listing row keyed by Trading 212 ticker.

### unclassified.csv

Header: `ticker,company_id,name,isin,reason`

Human review queue for rows missing useful theme/classification coverage.

### identity_issues.csv

Header: `issue_code,ticker,isin,security_id,company_id,name,reason,suggested_action`

Human review queue for identity collisions, missing identifiers, duplicate tickers, low confidence mappings, and manual override misses.

### enrichment_failures.csv

Header: `ticker,isin,name,provider,attempted_symbols,status,error,next_action`

Human review queue for enrichment cache/provider misses, stale or failed rows, ambiguous matches, and cache schema issues.

## Compatibility Expectations

Frontend code should tolerate missing optional fields and unknown additional fields. It should not depend on object key order. Existing array-shaped files must stay arrays unless the frontend is updated to support both old and new shapes.

Contract tests cover file presence, manifest metadata, checksum coverage, CSV headers, selected golden outputs, and the frontend-compatible `catalogue.json` shape.
